Here is a simple, step-by-step explanation of how the authentication works:

```
[1. User Logs In]
      |
      v (Phone gets a secure Google/Firebase "Login Ticket")
[2. Phone calls Cloudflare Worker]
      |
      +---> "Here is my Login Ticket. I want to upload a file."
      |
[3. Worker checks the Ticket]
      |
      +---> Checks Google's signature to prove the ticket is real and hasn't expired.
      +---> Reads the unique User ID (e.g., "User_12345").
      |
[4. Worker creates a Single-Use S3 Link]
      |
      +---> Link is locked to: private/User_12345/filename.db
      +---> Link automatically expires in 15 minutes.
      |
[5. Phone Uploads File]
      |
      v (Phone uploads directly to S3 using the temporary link)
[Railway S3 Storage]
```

### In simple terms:

1. **The Phone** does the login and gets a secure, tamper-proof "Ticket" from Google.
2. **The Cloudflare Worker** inspects that ticket using Google's public key (like a customs officer checking a passport hologram).
3. **The S3 Bucket** receives the file via a temporary link that the Worker signed.

This keeps your master S3 credentials safe on Cloudflare, ensures users can only access their own files, and works seamlessly with your Google Login flow.

Here is the technical explanation of how the verification works.

A "Login Ticket" (technically called a **JWT** or JSON Web Token) is a single text string divided into 3 parts separated by dots:

```text
Header  .  Payload  .  Signature
```

Let's look at what each part contains and how the Worker checks them:

---

### Part 1: The Payload (The Info)

This is simple text (encoded in Base64) containing user information:

- `user_id`: "User_12345"
- `exp` (Expiration Time): "2026-05-23T12:00:00"
- `iss` (Issuer): "https://securetoken.google.com/your-app"

### Part 2: The Signature (The Security Seal)

This is a cryptographic hash generated by Google/Firebase when the user logged in.

- Only Google has the **Private Key** (a secret mathematical formula) used to create this signature.
- Google publishes **Public Keys** (mathematical matches to their private keys) online for anyone to download.

---

### How the Cloudflare Worker Verifies it:

When the mobile app sends the ticket to the Cloudflare Worker, the Worker does 3 checks:

#### Check 1: Time Check

The Worker reads the `exp` (Expiration Time) in the Payload. If the current time is past the expiration time, the Worker rejects it immediately.

#### Check 2: Issuer Check

The Worker checks the `iss` (Issuer) field to ensure the ticket was issued by Google/Firebase for your specific app, and not by some other app.

#### Check 3: Cryptographic Signature Check (The most important part)

1. The Worker fetches Google’s **Public Keys** (which are cached on Cloudflare for speed).
2. The Worker runs a cryptographic math function (`crypto.subtle.verify`) using **Google's Public Key**, the **Payload text**, and the **Signature**.
3. **If the math matches:** The Worker knows for a fact that Google signed this ticket, and it has not been modified.

---

### Why is this un-hackable?

If a malicious user attempts to hack the ticket by changing the `user_id` from `"User_12345"` to `"User_99999"` to access someone else's files:

- The payload text changes.
- When the Worker checks the signature using Google's public key, the math **will not match** because the signature was generated for the original payload text.
- The Worker instantly rejects the request as tampered.
