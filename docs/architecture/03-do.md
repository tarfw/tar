# 03 — Durable Objects (Complete Guide)

Everything about how Cloudflare Durable Objects work and how TAR uses them.

## 1. What is a Durable Object?

A Durable Object (DO) is a **single-threaded, persistent, edge SQLite database with real-time push**.

```
┌─────────────────────────────────────────────────────────────┐
│                    Durable Object                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  SQLite Database                                     │   │
│  │  - Full SQL (SELECT, INSERT, UPDATE, DELETE)         │   │
│  │  - Persistent (survives restarts, crashes)           │   │
│  │  - Up to 10 GB per DO                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Single-Threaded Execution                           │   │
│  │  - Only ONE request runs at a time                   │   │
│  │  - No locks, no deadlocks                            │   │
│  │  - Automatic queuing                                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  WebSocket Support                                   │   │
│  │  - Real-time push to connected clients               │   │
│  │  - Hibernation (idle connections ~free)               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Alarm API                                           │   │
│  │  - Scheduled tasks (e.g., daily compaction)          │   │
│  │  - Runs even when DO is idle                         │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### The Problem DO Solves

```
Without DO:                    With DO:
┌─────┐    ┌─────┐            ┌─────┐    ┌─────┐
│  A  │───▶│  DB │            │  A  │───▶│     │
└─────┘    └──┬──┘            └─────┘    │ DO  │
             △                          │     │
┌─────┐    ┌──┴──┐            ┌─────┐    │Queue│
│  B  │───▶│     │            │  B  │───▶│[A,B]│
└─────┘    └─────┘            └─────┘    └─────┘
         Race condition!              No race condition
```

## 2. DO Lifecycle

```
┌─────────────────────────────────────────────────────────────┐
│                    DO Lifecycle                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. CREATION                                                │
│     Client calls: env.SYNC.idFromName("s:store_101")       │
│     → Cloudflare creates DO with unique ID                  │
│     → DO spins up, initializes SQLite                       │
│                                                             │
│  2. ACTIVE                                                  │
│     Client connects via WebSocket                           │
│     → DO handles requests                                   │
│     → SQLite reads/writes                                   │
│     → Broadcasts to other clients                           │
│                                                             │
│  3. HIBERNATION                                             │
│     No activity for ~30 seconds                             │
│     → WebSocket connections paused (not closed)             │
│     → SQLite flushed to disk                                │
│     → DO unloaded from memory                               │
│     → Cost: ~free                                           │
│                                                             │
│  4. WAKE-UP                                                 │
│     New request arrives                                     │
│     → DO spins up again                                     │
│     → SQLite loaded from disk                               │
│     → WebSocket connections resumed                         │
│     → Client doesn't notice any disconnect                  │
│                                                             │
│  5. DELETION (optional)                                     │
│     DO calls: state.storage.deleteAll()                     │
│     → SQLite destroyed                                      │
│     → DO removed                                            │
│     → Cost: zero                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────┐
│                    Request Flow                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. Client App                                              │
│     │                                                       │
│     │  POST /api/sync                                       │
│     │  Header: Authorization: Bearer <jwt>                  │
│     │  Query: scope=s:store_101                             │

│     │                                                       │
│     ▼                                                       │

│     │                                                       │

│     │                                                       │
│     ▼                                                       │
│  4. DO SQLite                                               │
│     │                                                       │
│     │  SELECT * FROM matter WHERE type = 'stock'            │
│     │  INSERT INTO motion ...                               │
│     │                                                       │
│     ▼                                                       │
│  5. Response                                                │
│     │                                                       │
│     │  JSON: { motion: [...], form: [...], matter: [...] }  │
│     │                                                       │
│     ▼                                                       │
│  6. WebSocket Push (to other clients)                       │
│     │                                                       │
│     │  Broadcast: { type: "new-sale", amount: 500 }         │
│     │                                                       │
└─────────────────────────────────────────────────────────────┘
```

### Worker Gateway Code

```javascript
// worker-gateway.js
export default {
  async fetch(request, env) {
    const url = new URL(request.url);
    const scopeCode = url.searchParams.get("scope"); // "s:store_101"
    const token = request.headers.get("Authorization")?.split(" ")[1];

    try {
      // 1. Verify JWT
      const payload = await verifyJwt(token, env.JWT_SECRET);
      
      // 2. Check scope permission
      if (!payload.scopes.includes(scopeCode)) {
        return new Response("Unauthorized Scope", { status: 403 });
      }
      
      // 3. Get DO by name (deterministic)
      const doId = env.SYNC_DO.idFromName(scopeCode);
      const doStub = env.SYNC_DO.get(doId);
      
      // 4. Forward request to DO
      const modified = new Request(request, {
        headers: { 
          ...Object.fromEntries(request.headers), 
          "X-User-Id": payload.userId 
        }
      });
      
      return doStub.fetch(modified);
      
    } catch {
      return new Response("Invalid Token", { status: 401 });
    }
  }
};
```

## 5. DO Internal Structure

### SQLite Schema

```sql
-- TAR's 4-table schema (inside each DO)

-- 1. form: Blueprint/template
CREATE TABLE form (
  id TEXT PRIMARY KEY,
  code TEXT,
  type TEXT NOT NULL,
  title TEXT,
  public INT DEFAULT 0,
  active INT DEFAULT 1,
  data TEXT,  -- JSON
  time TEXT
);

-- 2. matter: Realized instance
CREATE TABLE matter (
  id TEXT PRIMARY KEY,
  form TEXT,  -- FK → form.id
  type TEXT,
  qty REAL,
  value REAL,
  active INT DEFAULT 1,
  geo TEXT,
  start TEXT,
  end TEXT,
  data TEXT,  -- JSON
  time TEXT
);

-- 3. motion: Event ledger (append-only)
CREATE TABLE motion (
  stream TEXT,
  seq INT,
  action INT NOT NULL,
  phase INT,
  delta REAL,
  client_ref TEXT,
  data TEXT,  -- JSON
  time TEXT,
  PRIMARY KEY (stream, seq)
);

-- 4. bond: Graph links
CREATE TABLE bond (
  src TEXT,
  tgt TEXT,
  type TEXT,
  weight REAL,
  active INT DEFAULT 1,
  time TEXT,
  PRIMARY KEY (src, tgt, type)
);
```

## 6. DO Request Handling

### Fetch Handler (HTTP Requests)

```javascript
// Inside DO: tar-do.js
export class TarDO {
  constructor(state, env) {
    this.state = state;
    this.env = env;
    this.sql = state.storage.sql;
  }

  async fetch(request) {
    const url = new URL(request.url);
    
    // Route to handlers
    if (url.pathname === "/api/sync") {
      return this.handleSync(request);
    }
    if (url.pathname === "/api/query") {
      return this.handleQuery(request);
    }
    if (url.pathname === "/kick") {
      return this.handleKick(request);
    }
    
    return new Response("Not Found", { status: 404 });
  }

  async handleSync(request) {
    const { last_synced_seq, last_synced_time } = await request.json();
    
    // Get changes since last sync
    const motion = this.sql.exec(
      "SELECT * FROM motion WHERE seq > ?", 
      last_synced_seq
    ).all();
    
    const form = this.sql.exec(
      "SELECT * FROM form WHERE time > ?", 
      last_synced_time
    ).all();
    
    const matter = this.sql.exec(
      "SELECT * FROM matter WHERE time > ?", 
      last_synced_time
    ).all();
    
    const bond = this.sql.exec(
      "SELECT * FROM bond WHERE time > ?", 
      last_synced_time
    ).all();
    
    return Response.json({ motion, form, matter, bond });
  }
}
```

### WebSocket Handler (Real-time)

```javascript
// Inside DO: tar-do.js
export class TarDO {
  async fetch(request) {
    // ... other routes ...
    
    // WebSocket upgrade
    if (request.headers.get("Upgrade") === "websocket") {
      return this.handleWebSocket(request);
    }
  }

  async handleWebSocket(request) {
    const pair = new WebSocketPair();
    const [client, server] = Object.values(pair);
    
    // Accept WebSocket
    this.state.acceptWebSocket(server);
    
    return new Response(null, { 
      status: 101, 
      webSocket: client 
    });
  }

  async webSocketMessage(ws, message) {
    const data = JSON.parse(message);
    
    // Handle different message types
    if (data.type === "sale") {
      await this.handleSale(data, ws);
    }
    if (data.type === "query") {
      await this.handleQuery(data, ws);
    }
  }

  async handleSale(data, ws) {
    // 1. Write to SQLite (single-threaded, no race conditions)
    this.sql.exec(
      "INSERT INTO motion (stream, seq, action, delta, data) VALUES (?, ?, ?, ?, ?)",
      data.stream, data.seq, 201, data.amount, JSON.stringify(data)
    );
    
    // 2. Broadcast to all connected clients
    this.state.getWebSockets().forEach(client => {
      client.send(JSON.stringify({
        type: "new-sale",
        amount: data.amount,
        timestamp: Date.now()
      }));
    });
  }
}
```

```
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. DO is ACTIVE                                            │
│     ┌─────────────────────────────────────────────────┐    │
│     │  In Memory:                                     │    │
│     │  - SQLite database (hot data)                   │    │

│     ┌─────────────────────────────────────────────────┐    │
│     │  On Disk:                                       │    │

│     │                                                 │    │

│     └─────────────────────────────────────────────────┘    │
│                                                             │
│  4. NEW REQUEST ARRIVES                                     │
│     │                                                       │

│                                                             │
│  5. DO WAKES UP                                             │
│     ┌─────────────────────────────────────────────────┐    │
│     │  - SQLite loaded from disk                      │    │
│     │  - WebSocket connections resumed                │    │

│     │                                                 │    │
```

### Hibernation API Code

```javascript
// Inside DO: tar-do.js
export class TarDO {
  constructor(state, env) {
    this.state = state;
    
    // Use Hibernatable WebSocket API
    this.state.blockWebSocketDuringMessageProcessing();
  }

  async webSocketMessage(ws, message) {
    // Process message
    const data = JSON.parse(message);
    await this.processMessage(data);
    
    // WebSocket stays open, but DO can hibernate after
    // No need to keep connection alive manually
  }

  async webSocketClose(ws, code, reason) {
    // Clean up when client disconnects
    this.connectedClients.delete(ws);
  }

  async alarm() {
    // Runs even when DO is idle (via Alarm API)
    // Perfect for scheduled tasks like compaction
    await this.compactMotion();
  }
}
```

## 8. Single-Threaded Execution

```
Scenario: Two cashiers tap "Sell" at the same time

Without DO (Regular Database):
┌─────────┐         ┌─────────┐
│Cashier A│──┐      │Cashier B│──┐
└─────────┘  │      └─────────┘  │
             ▼                   ▼
        ┌─────────────────────────────┐
        │      Regular Database        │
        │  ┌─────────────────────┐   │
        │  │ Both try to:        │   │
        │  │ 1. Read stock=25    │   │  ← Race condition!
        │  │ 2. Write stock=24   │   │
        │  │ Result: stock=24    │   │  ← Should be 23!
        │  └─────────────────────┘   │
        └─────────────────────────────┘

With DO:
┌─────────┐         ┌─────────┐
│Cashier A│──┐      │Cashier B│──┐
└─────────┘  │      └─────────┘  │
             ▼                   ▼
        ┌─────────────────────────────┐
        │      Durable Object         │
        │  ┌─────────────────────┐   │
        │  │ Request Queue:      │   │
        │  │ [A, B]              │   │
        │  │                     │   │
        │  │ Run A:              │   │
        │  │ 1. Read stock=25    │   │
        │  │ 2. Write stock=24   │   │
        │  │                     │   │
        │  │ Run B:              │   │
        │  │ 1. Read stock=24    │   │  ← Correct!
        │  │ 2. Write stock=23   │   │
        │  └─────────────────────┘   │
        └─────────────────────────────┘
```

### Code Example

```javascript
// Inside DO: tar-do.js
export class TarDO {
  async handleSale(data) {
    // This code runs ONE AT A TIME
    // No two requests can execute this simultaneously
    
    // 1. Read current stock (safe, no lock needed)
    const stock = this.sql.exec(
      "SELECT qty FROM matter WHERE id = ?", 
      data.stockId
    ).one();
    
    // 2. Check if enough stock
    if (stock.qty < data.quantity) {
      return { error: "Insufficient stock" };
    }
    
    // 3. Deduct stock (safe, single-threaded)
    this.sql.exec(
      "UPDATE matter SET qty = qty - ? WHERE id = ?",
      data.quantity, data.stockId
    );
    
    // 4. Log the sale
    this.sql.exec(
      "INSERT INTO motion (stream, seq, action, delta) VALUES (?, ?, ?, ?)",
      data.stockId, this.nextSeq(), 101, -data.quantity
    );
    
    // 5. Broadcast to other clients
    this.broadcast({
      type: "stock-updated",
      stockId: data.stockId,
      newQty: stock.qty - data.quantity
    });
    
    return { success: true };
  }
  
  nextSeq() {
    // DO holds seq in-memory, hands out clean integers
    // No two requests get the same seq
    return ++this.currentSeq;
  }
}
```

## 10. Alarm API (Scheduled Tasks)

```javascript
// Inside DO: tar-do.js
export class TarDO {
  async alarm() {
    // Runs at scheduled time (e.g., 3:00 AM)
    await this.compactMotion();
  }

  async compactMotion() {
    // 1. Get cutoff date (7 days ago)
    const cutoff = new Date(Date.now() - 7 * 24 * 60 * 60 * 1000).toISOString();
    
    // 2. Export old motion to R2
    const oldMotion = this.sql.exec(
      "SELECT * FROM motion WHERE time < ?", 
      cutoff
    ).all();
    
    if (oldMotion.length > 0) {
      // Upload to R2
      await this.env.R2.put(
        `motion/${this.state.id.toString()}/${cutoff}.json`,
        JSON.stringify(oldMotion)
      );
      
      // 3. Delete from SQLite
      this.sql.exec(
        "DELETE FROM motion WHERE time < ?", 
        cutoff
      );
      
      // 4. VACUUM to reclaim space
      this.sql.exec("VACUUM");
    }
    
    // 5. Schedule next compaction
    this.state.setAlarm(Date.now() + 24 * 60 * 60 * 1000); // 24 hours later
  }
}
```

## 11. Best Practices

### DO Design

| Practice | Why |
|:---------|:----|
| One DO per scope | Isolates data, prevents contention |
| Use `state.storage.sql` | Direct SQLite access, faster than ORM |
| Minimize state in memory | DO can hibernate, wake up fresh |
| Use Alarm for cleanup | Scheduled tasks run even when idle |

### WebSocket

| Practice | Why |
|:---------|:----|
| Use Hibernatable API | Idle connections ~free |
| Broadcast sparingly | Only send to relevant clients |
| Handle disconnections | Clean up on `webSocketClose` |
| Validate JWT on connect | Prevent unauthorized access |

### SQLite

| Practice | Why |
|:---------|:----|
| Use prepared statements | Faster, prevents SQL injection |
| Batch inserts | Reduce round trips |
| Index frequently queried columns | Faster reads |
| Use transactions for multi-step writes | Atomicity |

```
───────────────────────────────────
Client reads menu → DO serves from SQLite
Client reads stock → DO serves from SQLite
Client writes sale → DO writes to SQLite, broadcasts

Frequency: 90% reads, 10% writes
Optimization: Cache in memory, hibernate between requests


Pattern 2: Write-Heavy (Order)
──────────────────────────────
Customer places order → DO creates order
Restaurant confirms → DO updates phase
Kitchen prepares → DO updates phase
Driver delivers → DO updates phase

Frequency: 100% writes (phase updates)
Optimization: Phase updates are in-place (cheap)


Pattern 3: Coordination (Checkout Lock)
────────────────────────────────────────
Two customers buy last item → DO serializes requests
Customer A wins → stock decremented
Customer B loses → "Out of stock" response

Frequency: Low (high contention scenarios)
Optimization: Single-threaded = no locks needed
```

## 13. Debugging

| Issue | Cause | Solution |
|:------|:------|:---------|
| DO not waking up | Request not hitting DO | Check Worker routing |
| WebSocket disconnects | JWT expired | Re-authenticate |
| Sync conflicts | Clock skew | Use LWW, trust server time |
| High latency | DO cold start | First request after hibernation is slower |
| SQLite locked | Too many concurrent writes | Impossible in single-threaded DO |

## 14. Summary

| Concept | What It Means |
|:--------|:--------------|
| **Durable** | Survives restarts, crashes, deployments |
| **Object** | One instance per scope (s:store_101) |
| **Single-threaded** | No race conditions, automatic queuing |
| **SQLite** | Full SQL database, not key-value |
| **WebSocket** | Real-time push to connected clients |
| **Hibernation** | Idle DOs cost ~free |
| **Edge** | Runs on Cloudflare's edge network (low latency) |

**The key insight:** DO gives you a **single-threaded, persistent, edge SQLite database with real-time push** — perfect for collaborative apps where multiple users edit the same data.
